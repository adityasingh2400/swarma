## Problem Statement

ReRoute v1 uses Fetch AI uAgents for item research and marketplace posting. Three problems:

1. **Marketplace coverage is fake.** eBay has a real API integration. Facebook, Mercari, Depop are stubs. Adding each new platform requires building and maintaining an API connector, handling OAuth, and dealing with API changes. This doesn't scale.

2. **Latency is bad.** Video upload + sequential frame extraction + batch Gemini analysis creates a long wait before anything happens. The pipeline feels slow even though the route bidding (asyncio.gather) is concurrent.

3. **Agent work is invisible.** The v1 design doc (March 22) identified this: the backend does impressive concurrent work that the frontend hides. v1 added a Mission Control grid and WebSocket events, but agents still operate as black boxes. You see status badges, not what agents are actually doing.

Browser-Use solves all three: browser automation replaces API connectors (any website is a platform), concurrent browser agents are visually observable (you can literally see the browser), and streaming frame analysis with immediate agent dispatch kills the batch latency.

## What Makes This Cool

You record a 30-second video of stuff you want to sell. Within seconds, 12-15 browser windows light up on your screen. You watch one agent searching eBay sold listings. Another is navigating Facebook Marketplace's listing form. Another is checking Apple's trade-in page. Another is browsing Mercari for competitive pricing. All simultaneously. All visible. All real.

Every agent streams live at 5 fps — you see all of them at once, simultaneously, in a grid. Watch one type a listing title on eBay while another uploads photos to Facebook Marketplace while another sets a price on Mercari. The system isn't a dashboard showing you status badges. It's a live mission control showing you REAL BROWSERS doing REAL WORK, all at the same time.

Nobody has built this. Cross-listing tools (Vendoo, Nifty) work behind the scenes. ReRoute v2 makes the intelligence visible.

## Cross-Model Perspective

An independent Claude subagent reviewed the session cold. Key insights:

- **"Spectacle at scale" is the real goal.** "They don't want a tool that works. They want a tool that makes people's jaws drop watching it work. The browser feeds aren't a feature request; they're a stage direction."

- **The 50% you get free vs the 50% you build:** Browser-Use + Playwright gives browser control, parallel agents, CDP protocol access. You build: the orchestrator (spawning right agents per item), the real-time visualization pipeline (CDP screenshots → WebSocket → frontend), marketplace playbooks, and the video-to-task pipeline.
- **Weekend build order:** (1) One reliable eBay listing agent from structured JSON, (2) WebSocket frame streamer with 4 parallel agents, (3) Wire video intake to agent spawning, (4) Add second platform + polish.

## Approaches Considered

### Approach A: "Live Swarm" — Maximum Visual Impact
12-15 concurrent Browser-Use agents with real-time CDP video streaming to a React grid. Every agent's browser visible simultaneously at 5 fps minimum. Video → streaming Gemini analysis → agents spawn as items are identified.
- Effort: L | Risk: High | Pros: Maximum "whoa," most technically impressive | Cons: Resource-intensive, needs careful memory management

### Approach B: "Cinematic Orchestrator" — Speed + Polish
Same concurrent agents, but key-moment screenshots instead of live streaming. Animated card grid with status, latest screenshot, progress bars.
- Effort: M | Risk: Low | Pros: Reliable, polished | Cons: Not "truly live"

### Approach C: "Hybrid Theater" — Best of Both
Orchestration grid as default + single-agent focus mode with live feed. Replay mode for presentation reliability.
- Effort: L | Risk: Medium | Pros: Balanced | Cons: More complex

## Recommended Approach

**Approach A: Live Swarm** — chosen by the user. No compromises, no replay fallbacks. The real thing, live, all at once. Every agent streams at 5 fps minimum, all simultaneously displayed. No focus mode, no click-to-drill-in — the whole grid is always live.

### Authentication Strategy

Agents need to be logged in to post listings. Approach:

1. **Pre-authenticated browser profiles**: Before the demo, manually log in to eBay, Facebook, Mercari, and Depop in separate Chromium profiles. Save these profiles to disk (`~/.reroute/profiles/{platform}/`).
2. **Profile loading on startup**: When the warm browser context pool launches, each context loads the appropriate profile for its target platform via Playwright's `user_data_dir` parameter.
3. **Session persistence**: Profiles persist cookies/sessions between runs. If a session expires, re-authenticate manually (acceptable for hackathon, automate later).
4. **Browser-Use Cloud alternative**: Their `profile.sh` sync can push auth profiles to remote browsers. Use this if switching to cloud.

### Failure Handling

Agents will fail. Platforms change, CAPTCHAs appear, elements move. Strategy:

- **Agent retry**: On failure, retry once with a 3-second delay. Browser-Use's LLM-driven error recovery handles most transient issues (wrong selector, page not loaded).
- **UI failure state**: Failed agents show a red border + error icon in the grid. The browser feed freezes on the last successful screenshot with an error overlay.
- **Graceful degradation**: If a platform is unreachable (blocked, down, CAPTCHA wall), that agent is marked "BLOCKED" and the pipeline continues with remaining platforms. Success criteria: at least 2 of 4 platforms must succeed.
- **No cascading failures**: Each agent runs in an isolated browser context. One agent crashing does not affect others.
- **CAPTCHA handling**: For hackathon, run in headful mode (visible browser windows are the whole point). If a CAPTCHA appears, the presenter can manually solve it live during the demo, which actually makes the demo MORE impressive ("even when it hits a CAPTCHA, the system handles it"). For production: integrate 2Captcha or Anti-Captcha service.
- **Demo insurance**: Before demo day, run the full pipeline 5 times. Identify which platforms are reliable vs flaky. Lead with the reliable ones in the demo.

### Agent Architecture: Hybrid Structured Workflows

Each marketplace agent uses a HYBRID approach (per second opinion):

```python
# NOT this (flaky):
agent = Agent(task="Post this iPhone 15 on eBay for $800", llm=llm, browser=browser)

# THIS (reliable):
agent = Agent(
    task=f"""
    Go to ebay.com/sell. Click 'List an item'.
    In the title field, type: {item.title}
    Select category: {item.category}
    Set condition: {item.condition}
    Upload these images: {item.image_paths}
    Set price to: ${item.price}
    Select shipping: Standard Shipping
    Click 'List it'
    """,
    llm=llm,
    browser=browser,
    tools=marketplace_tools,  # custom tools for image upload handling etc.
)
```

Each platform gets a structured playbook (a Python function that generates the detailed task string from item data). Natural language handles navigation and error recovery. Structured data handles form filling.

### Marketplace Playbooks (per platform)

**eBay** (most reliable, best-tested):
- Entry: `ebay.com/sell` → "List an item"
- Fields: Title (80 char max), Category (search-select), Condition (dropdown), Item Specifics (varies by category), Photos (drag-drop or file input), Price (fixed or auction), Shipping (calculated or flat)
- Gotchas: Category selection is a search-and-select UI, not a simple dropdown. Image upload uses a file input that Playwright's `set_input_files()` handles. eBay may require phone verification for new accounts.

**Facebook Marketplace**:
- Entry: `facebook.com/marketplace/create/item`
- Fields: Photos, Title, Price, Category (dropdown), Condition (dropdown), Description, Location (auto-filled from profile)
- Gotchas: Requires location (auto-fills from logged-in profile). Photo upload is drag-and-drop, fall back to file input. May prompt for Marketplace Terms acceptance on first listing. No shipping option for local-only listings.

**Mercari**:
- Entry: `mercari.com/sell`
- Fields: Photos (up to 12), Title, Description, Category (hierarchical), Brand (search-select), Condition (radio buttons), Price, Shipping (prepaid label or ship on your own)
- Gotchas: Mercari's category selection is 3-level hierarchical (Category → Subcategory → Sub-subcategory). Smart shipping requires weight/dimensions. Photo upload order matters (first photo = thumbnail).

**Depop**:
- Entry: `depop.com/products/create`
- Fields: Photos (up to 4), Description, Category, Brand, Size, Condition, Price, Shipping
- Gotchas: Depop is fashion-focused, requires Size and Brand fields. Description is the primary search field (no separate title). Photo limit is only 4.

### Image Pipeline: Video to Listing-Ready Photos

Between item identification and listing agent dispatch:

1. **Frame selection**: When Gemini identifies an item, it also returns which frame indices show that item best (clearest view, least blur, best angle). The orchestrator selects the top 3-5 frames per item.
2. **Image extraction**: Selected frames are extracted from the video at full resolution using ffmpeg (`-vf select='eq(n,FRAME_NUM)'`).
3. **Image preparation**: Crop to item bounds using Gemini's bounding box output (prompt Gemini with "return bounding_box as [x1,y1,x2,y2] normalized 0-1 for each item"). If bounding boxes are unavailable or imprecise, fall back to full-frame images (still usable, just less polished). Resize to marketplace-optimal dimensions (1600x1600 for eBay, 1080x1080 for others), apply sharpening.
4. **File staging**: Prepared images saved to `./data/listing-images/{item_id}/` as JPEG files. These paths become `item.image_paths` used by listing agents.
5. **Playwright upload**: Listing agents use `page.set_input_files()` to upload staged images to marketplace file inputs.

### Route Decision Logic

After all research agents complete for an item, the Route Decision runs locally (no browser needed):

- **Input**: Research results from all 5 agents (eBay sold prices, FB Marketplace comps, Mercari prices, Apple trade-in value, repair cost + parts availability)
- **Algorithm**: Weighted scoring same as v1: 45% value, 25% confidence, 15% effort, 15% speed. Each platform scored 0-100.
- **Output**: Ranked list of platforms to list on + recommended price per platform. Top 3-4 platforms get listing agents dispatched.
- **Implementation**: Single Gemini call with research results as context, returning structured JSON with platform rankings and prices. Fast (~1-2s), no browser needed.
- **Definitions**: Value = expected sale price minus fees. Confidence = data quality (how many comps found, price variance). Effort = post-listing work for the seller (shipping complexity, buyer communication likelihood). Speed = estimated time-to-sale based on platform activity and demand signals.

### Latency Solution: Streaming Pipeline

```
CURRENT (v1):                    NEW (v2):
Video upload ──────── 5s         Video upload ──────── 5s
Frame extraction ──── 8s         Frame 1 extracted ─── 0.5s
  (wait for all)                   → Agent spawned immediately
Gemini analysis ───── 4s         Frame 2 extracted ─── 1.0s
  (wait for all items)             → Another agent spawned
Agent dispatch ────── 0.1s       Frame 5 extracted ─── 2.5s
Route bidding ─────── 6s           → More agents spawned
                                 
Total wait: ~23s                 First agent working: ~3s
                                 All agents working: ~8s
```

The key: don't wait for all frames. Extract frames in a streaming fashion (ffmpeg piped output), send each frame to Gemini immediately, dispatch agents as soon as items are identified. The pipeline is fully streaming, not batch.

**Qualification**: The first 1-3 frames may not contain identifiable items (hand motion, camera adjusting). Realistically, Gemini needs a small batch of 3-5 frames (covering ~6-10 seconds of video) to reliably identify the first item. The "First agent working: ~3s" estimate assumes the video starts with a clear item shot. Conservative estimate with camera adjustment: first agent at ~5-8s. Still dramatically faster than v1's 23s.

**Gemini error handling**: If a Gemini call fails or times out during streaming analysis, retry once with a 2s delay. If the retry fails, skip that frame batch and continue with the next. Log the failure but don't block the pipeline. If Gemini returns no items from a frame batch, that's normal (not every frame shows an item), just continue.

### Concurrency Management

- **Browser context pool**: Pre-launch browser contexts when FastAPI server starts (warm pool, before any user interaction). Target: 12-15 active contexts. Must validate on target hardware first (see Step 0 in Next Steps). Realistic ceiling on 16GB MacBook Pro may be 10-12, use 32GB machine if available.
- **Agent queue**: With 2 items producing up to 18 agents, 3-6 will be queued at any time. Priority: research agents first (they unblock the route decision), then listing agents. Item 1's research agents take priority over item 2's if contexts are scarce.
- **Memory budget**: Each context with an active marketplace page (heavy JS) realistically uses 200-400MB. At 10 contexts: ~2-4GB. At 15: ~3-6GB. Plus Chromium shared process (~500MB-1GB), Python backend, ffmpeg. On a 16GB machine, 10-12 contexts is the safe ceiling. On 32GB, 15 is comfortable. Step 0 benchmarking will establish the real number.
- **Live video streaming (capture vs delivery)**:
  - **Capture**: Each agent's browser streams via CDP `Page.startScreencast` — event-driven push, no polling. Browser pushes JPEG frames on every N-th vsync. At 5 fps: `everyNthFrame=12`. 5 fps x 12 agents = 60 frames/sec inbound total — significant CPU load; validate with Step 0 benchmarking and tune `everyNthFrame` if needed, but 5 fps is the hard floor.
  - **Delivery rate to frontend**: All agents delivered uniformly at 5 fps minimum. No per-agent throttle differentiation — every feed in the swarm stays live simultaneously. There is no focused agent concept.
  - **Resolution**: All agents rendered at 320x240 JPEG quality 60 (~15-25KB each). Single resolution tier — no focus/grid split.
  - **Transport**: Binary WebSocket frames (not base64 JSON) for video stream frames to avoid 33% overhead. Status events remain JSON text frames.
  - **Bandwidth estimate**: 12 agents at 5 fps x 20KB = 1,200KB/s (~1.2MB/s) sustained. Comfortable on localhost; validate on demo hardware during Step 0.
